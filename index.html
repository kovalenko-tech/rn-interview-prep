Q1Объясни разницу Bridge и JSI — как будто я не знаю▾

Bridge (старая архитектура) — это очередь сообщений между JS и Native. Каждый вызов сериализуется в JSON, передаётся асинхронно, десериализуется на другой стороне. Это как REST API, только внутри приложения.

JSI (JavaScript Interface) — C++ прослойка, которая даёт JS прямой доступ к нативным объектам через указатели. Никакого JSON, никакой асинхронности. JS может вызвать нативный код синхронно и получить результат сразу.

💡 Аналогия: Bridge — звонок по телефону с переводчиком. JSI — разговор напрямую без переводчика.

Почему это важно:

  • Синхронные вызовы — можно получить размер view до рендера
  • Нет накладных расходов сериализации — быстрее при большом количестве событий
  • Открывает путь к Concurrent React (Suspense, transitions)

Q2Что такое Hermes и зачем Meta его создала?▾

Hermes — JavaScript движок разработанный Meta специально для мобильных приложений на React Native.

Проблема с обычными движками (V8, JSC): они используют JIT-компиляцию — JS парсится и компилируется при запуске приложения. На слабых Android устройствах это медленно.

Решение Hermes:

  • AOT компиляция — JS → bytecode происходит при сборке (не при запуске)
  • На устройстве уже готовый bytecode → парсинг не нужен → TTI сокращается на 30–40%
  • Меньше потребление RAM — GC оптимизирован под мобильные
  • Меньший размер бинарника чем V8
⚠️ Минус: Hermes без JIT — повторно вызываемый код не оптимизируется так хорошо как в V8. На практике для мобильных приложений это незаметно.

Q3Что такое Fabric и зачем его переписали?▾

Fabric — новый рендерер React Native, замена старого UIManager.

Старый рендерер: Shadow Tree жил в JS → через Bridge → Shadow Thread в C++ → UI Thread. Три копии данных, всё асинхронно.

Fabric: Shadow Tree живёт в C++ и доступен синхронно из JS через JSI. Данные копируются меньше.

Что это даёт:

  • Поддержка Concurrent React — Suspense, transitions, startTransition
  • Синхронный layout — можно измерить размер view до следующего кадра
  • Приоритизация обновлений (urgent vs не-urgent)
  • Меньше потребление памяти

Q4TurboModules — что это и чем лучше старых Native Modules?▾

Старые Native Modules: все инициализируются при старте приложения, даже если не используются. Общение через Bridge (JSON).

TurboModules:

  • Lazy initialization — модуль загружается только при первом вызове
  • JSI-based — прямой вызов без JSON сериализации
  • Codegen — из TypeScript спецификации генерируется типобезопасный C++ интерфейс
  • Возможность синхронных вызовов
// Спецификация TurboModule
import type { TurboModule } from 'react-native';
import { TurboModuleRegistry } from 'react-native';

export interface Spec extends TurboModule {
  // синхронный вызов — возможен только через JSI
  multiply(a: number, b: number): number;
}

export default TurboModuleRegistry.getEnforcing<Spec>('Calculator');

Q5useEffect — все случаи вызова и типичная ловушка с closure▾ 
// 1. Без deps — каждый рендер
useEffect(() => { document.title = title; });

// 2. [] — один раз при монтировании
useEffect(() => { fetchData(); }, []);

// 3. С deps — при изменении userId
useEffect(() => { fetchUser(userId); }, [userId]);

// 4. Cleanup — отписка, отмена запроса
useEffect(() => {
  const sub = subscribe(topic);
  return () => sub.unsubscribe(); // вызывается перед размонтированием
}, [topic]);

Closure trap — классическая ошибка:

// ❌ count "застрял" в замыкании — всегда 0+1=1
useEffect(() => {
  const t = setInterval(() => setCount(count + 1), 1000);
  return () => clearInterval(t);
}, []); // count не в deps!

// ✅ функциональное обновление — всегда актуальный prev
useEffect(() => {
  const t = setInterval(() => setCount(prev => prev + 1), 1000);
  return () => clearInterval(t);
}, []);
💡 Правило: если используешь переменную из внешнего скопа внутри useEffect — она должна быть в deps, либо используй функциональное обновление setState.

Q6useMemo vs useCallback — разница и когда они ВРЕДЯТ▾ 
// useMemo — кэширует РЕЗУЛЬТАТ вычисления
const sorted = useMemo(
  () => items.filter(i => i.active).sort(byDate),
  [items] // пересчёт только если items изменился
);

// useCallback — кэширует ФУНКЦИЮ
// (эквивалентно useMemo(() => fn, deps))
const handlePress = useCallback(
  () => navigate('Details', { id }),
  [id]
);

Когда нужны:

  • useMemo: тяжёлые вычисления (sort/filter 1000+ элементов)
  • useCallback: передаёшь callback в компонент с React.memo

Антипаттерны (когда вредят):

// ❌ overhead > benefit для простого умножения
const doubled = useMemo(() => count * 2, [count]);

// ❌ компонент не обёрнут в React.memo — useCallback бесполезен
<Button onPress={useCallback(() => doSomething(), [])} />

// ❌ объект в deps — новая ссылка каждый рендер = memo никогда не сработает
const result = useMemo(() => heavy(obj), [{ ...obj }]);
⚠️ Правило: сначала профилируй, потом оптимизируй. Преждевременная мемоизация — враг читаемости кода.

Q8Напиши хук useDebounce▾ 
function useDebounce<T>(value: T, delayMs: number): T {
  const [debounced, setDebounced] = useState<T>(value);

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebounced(value);
    }, delayMs);

    // cleanup: отменяем таймер при следующем изменении value
    return () => clearTimeout(timer);
  }, [value, delayMs]);

  return debounced;
}

// Использование для поиска
const [query, setQuery] = useState('');
const debouncedQuery = useDebounce(query, 500);

useEffect(() => {
  if (debouncedQuery) searchAPI(debouncedQuery);
}, [debouncedQuery]);
Q9Напиши useFetch с отменой запроса (memory leak fix)▾ 
function useFetch<T>(url: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    const controller = new AbortController();

    setLoading(true);
    setError(null);

    fetch(url, { signal: controller.signal })
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json() as Promise<T>;
      })
      .then(json => {
        setData(json);
        setLoading(false);
      })
      .catch(err => {
        if (err.name !== 'AbortError') { // игнорируем отмену
          setError(err);
          setLoading(false);
        }
      });

    return () => controller.abort(); // cleanup при unmount или смене url
  }, [url]);

  return { data, loading, error };
}
🚫 Без cleanup: если компонент размонтируется пока запрос в полёте — setState вызывается на размонтированном компоненте → memory leak + предупреждение React.

Q11Как профилировать RN приложение? Пошагово.▾

Шаг 1 — React DevTools Profiler:

  • Записать сессию → видишь какой компонент рендерится и сколько раз
  • "Why did this render?" → какой prop/state вызвал ре-рендер
  • Длинные полосы = медленный рендер

Шаг 2 — Flipper (Performance plugin):

  • JS Thread FPS и UI Thread FPS — отдельно
  • JS FPS упал, UI норм → проблема в JS коде
  • Оба упали → нативная проблема или heavy layout

Шаг 3 — Platform-specific:

  • iOS: Xcode Instruments → Time Profiler, Allocations, Leaks
  • Android: Android Studio Profiler → CPU, Memory, Network
💡 Бюджет кадра: 16ms при 60fps, 8ms при 120fps. JS frame > 16ms → пользователь видит рывок.
⚠️ Частые находки: console.log в продакшне (−20% perf), JSON.parse(JSON.stringify()) в hot path, useSelector на большой объект целиком.

Q14Optimistic updates — как реализовать с TanStack Query?▾ 
const addItem = useMutation({
  mutationFn: api.addItem,

  onMutate: async (newItem) => {
    // 1. Отменяем текущие запросы
    await queryClient.cancelQueries({ queryKey: ['items'] });
    // 2. Сохраняем предыдущее состояние
    const previous = queryClient.getQueryData(['items']);
    // 3. Оптимистично обновляем UI
    queryClient.setQueryData(['items'], old => [...old, newItem]);
    return { previous };
  },

  onError: (_err, _item, context) => {
    // 4. При ошибке — откат
    queryClient.setQueryData(['items'], context?.previous);
  },

  onSettled: () => {
    // 5. Всегда инвалидируем кэш
    queryClient.invalidateQueries({ queryKey: ['items'] });
  },
});

Q18Conditional types и infer — объясни с примером▾ 
// infer — "вытащить" тип из другого типа
type ArrayItem<T> = T extends (infer Item)[] ? Item : never;
type StrItem = ArrayItem<string[]>; // string

// Unwrap Promise
type Awaited<T> = T extends Promise<infer R> ? R : T;
type ApiUser = Awaited<Promise<User>>; // User

// Discriminated Union — TypeScript сужает тип по полю status
type Result<T> =
  | { status: 'success'; data: T }
  | { status: 'error'; error: string; code: number };

function handle<T>(result: Result<T>) {
  if (result.status === 'success') {
    result.data; // T — TypeScript знает
  } else {
    result.error;  // string
    result.code;   // number
  }
}

import { renderHook, act } from '@testing-library/react-hooks';

describe('useCounter', () => {
  it('starts with initial value', () => {
    const { result } = renderHook(() => useCounter(5));
    expect(result.current.count).toBe(5);
  });

  it('increments', () => {
    const { result } = renderHook(() => useCounter(0));

  });
});

function useDebounce<T>(value: T, delay: number): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const t = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(t);
  }, [value, delay]);
  return debounced;
}
Medium useFetch с AbortController (без memory leak) Показать решение ▾ 
function useFetch<T>(url: string) {
  const [state, setState] = useState<{data:T|null;loading:boolean;error:Error|null}>(
    { data: null, loading: true, error: null }
  );
  useEffect(() => {
    const ctrl = new AbortController();
    setState(s => ({ ...s, loading: true, error: null }));
    fetch(url, { signal: ctrl.signal })
      .then(r => { if (!r.ok) throw new Error(`${r.status}`); return r.json(); })
      .then(data => setState({ data, loading: false, error: null }))
      .catch(e => { if (e.name !== 'AbortError') setState({ data: null, loading: false, error: e }); });
    return () => ctrl.abort();
  }, [url]);
  return state;
}
Medium Infinite scroll хук с pagination Показать решение ▾ 
function useInfiniteList<T>(fetchFn: (page: number) => Promise<T[]>, pageSize = 20) {
  const [items, setItems] = useState<T[]>([]);
  const [page, setPage] = useState(1);
  const [loading, setLoading] = useState(false);
  const [hasMore, setHasMore] = useState(true);

  const loadMore = useCallback(async () => {
    if (loading || !hasMore) return;
    setLoading(true);
    try {
      const next = await fetchFn(page);
      if (next.length < pageSize) setHasMore(false);
      setItems(prev => [...prev, ...next]);
      setPage(p => p + 1);
    } finally { setLoading(false); }
  }, [loading, hasMore, page, fetchFn, pageSize]);

  useEffect(() => { loadMore(); }, []); // первая загрузка
  return { items, loadMore, loading, hasMore };
}
Hard Race condition при быстрой смене userId Показать решение ▾

Проблема: меняешь userId быстро → 2 запроса в полёте → второй может прийти раньше первого → показываем устаревшие данные.

function useUser(userId: string) {
  const [user, setUser] = useState<User | null>(null);
  useEffect(() => {
    let isCurrent = true; // флаг актуальности запроса
    fetchUser(userId).then(data => {
      if (isCurrent) setUser(data); // игнорируем устаревший ответ
    });
    return () => { isCurrent = false; };
  }, [userId]);
  return user;
}

function useThrottle<T>(value: T, limitMs: number): T {
  const [throttled, setThrottled] = useState(value);
  const lastRan = useRef(Date.now());

  useEffect(() => {
    const remaining = limitMs - (Date.now() - lastRan.current);
    const t = setTimeout(() => {
      setThrottled(value);
      lastRan.current = Date.now();
    }, Math.max(0, remaining));
    return () => clearTimeout(t);
  }, [value, limitMs]);

  return throttled;
}
